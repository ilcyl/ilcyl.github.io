三、生存期模型 - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - 5

3.1 相关定义

3.1.1 项目生存期选择

  • 预测型: 瀑布模型，V模型(需求、方案明确) 提前进行大量的计划工作，然后一次性执行；执行是一个连续的过程。
  • 迭代型:(需求不明确，项目复杂性高、变更频繁) 允许对未完成的工作进行反馈，从而改进和修改该工作。
  • 增量型: 向客户提供各个已完成的，可能立即使用的可交付成果。
  • 敏捷型: 既有迭代，也有增量，便于完善工作，频繁交付。

3.1.2 敏捷模型(需求不固定，试探模式，价值驱动)

  • Scrum：目前使用率最高
  1. 3个角色：产品负责人、开发团队、敏捷教练
  2. 3个工件：Sprint待办事项列表、产品待办清单、符合“完成的定义”的产品增量
  3. 5个事件：Sprint计划会、每日Scrum站会、Sprint评审会、Sprint回顾会、Sprint(短时间盒)
  4. 5个价值观：开放、尊重、勇气、专注、承诺
  • XP：极限编程模型

针对业务需求和软件开发实践的规则

  • Kanban
  • DevOps：开发和运维工作紧密合作(企业文化支持)

3.1.3 敏捷软件开发生命周期

需求分析 → 设计 → 编程/CI → 测试

3.1.4 AI驱动的开发模式

AI分析工具 → AI驱动架构与设计 → AI编程助手和生成器 → AI测试工具

3.2 习题

  1. 下面哪种敏捷实践使用率最高：Scrum
  2. 项目初期，一个项目需求不明确的情况下，应避免采用哪种生存期模型：V模型

4.1.1 需求建模的基本方法

  • 使用简单的语法格式成为Gherkin：
  1. GIVEN
  2. WHEN
  3. THEN

4.1.7 用户故事

  • 用户故事地图：
  1. 横坐标：业务流程时间线，纵坐标：商业价值(高→低)
  2. 右侧的发布阶段，发布1为MVP(最小化可行产品)
  • 用户故事是对功能的简短、简单的描述。它们通常遵循一个简单的模板：
  1. As a
  2. I want
  3. so that

4.1.8 MoSCoW

  • DSDM（动态系统开发方法）推崇的一种优先排序方法。元素包括：
  1. 必须要有：必须实现的功能，否则系统将无法运行。
  2. 应该有：功能很重要，但如果出现时间或资源约束，可以省略这些功能。
  3. 可能有：功能可以增强系统的功能，但其交付时间不是很关键。
  4. 想要有：功能仅服务于有限的用户群体，并且不会产生与前面项目相同的业务价值。

4.1.9 影响地图

  • 模版：从左到右分别为业务目标，角色，影响，功能项

4.1.10 数据流图(DFD)

  • 展示系统中的数据是如何加工处理和流动

五、任务分解 - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - - 9

5.1 相关定义

5.1.1 WBS(任务分解结构，全部需求)

  • WBS组织并定义了整个项目的范围
  • WBS是面向交付成果的
  • WBS是对项目由粗到细的分解过程
  • 图表、清单、字典(标识号、名称、说明、描述、流程)形式

5.1.2 工作包(work packages)

  • 一个工作包是WBS的最低层次的可交付成果的集合
  • 工作包应当由唯一主体负责
  • 每个工作包都必须有一个提交物

5.1.3 分解方法

  • 类比、模版参照、自上而下、自下而上

5.1.4 敏捷项目任务分解

  • 基于story的任务分解
  • Story Map(渐进式需求)

5.1.5 Story

  • 用户故事的特点是它们可以在不同细节层次上编写

5.1.6 Epic

  • 大型用户故事通常被称为Epic
  • 因为一个Epic通常对于敏捷团队来说在一个迭代中完成太大，所以它被分解成多个较小的用户故事，然后才会进行工作

5.2 习题

  1. WBS提供了项目范围基线 ✅
  2. 一个工作包可以分配给另外一个项目经理去完成 ✅
  3. 自上而下的任务分解方法从特殊到一般的方向进行，首先定义一些特殊的任务，然后将这些任务组织起来，形成更高级别的WBS层 ❌
  4. WBS的最高层次的可交付成果是工作包 ✅
  5. 敏捷项目的一个Epics还可以继续分解为一些Story ✅
  6. WBS确定了项目的范围 ✅

6.1.5 功能点估算

  • 用系统的功能数量来测量其规模
  • 通过评估、加权、量化得出功能点
  • 功能点估算方法：IFPUG、NESMA、COSMIC

6.1.6 IFPUG

  • FP = UFP * TCF(UFP：未调整功能点计数，TCF：技术复杂度因子)
  • UFP：
  • 数据功能（Data Functions）： 即用户与系统交互时操作的数据。包括以下几种：
  1. 内部逻辑文件（ILF，Internal Logical File）： 系统内部存储的用户数据。
  2. 外部接口文件（EIF，External Interface File）： 外部系统传入的、被系统使用的文件。
  • 事务功能（Transaction Functions）： 即用户与系统交互的处理流程。包括以下几种：
  1. 外部输入（EI，External Input）： 用户向系统输入数据。
  2. 外部输出（EO，External Output）： 系统向外部输出的数据。
  3. 外部查询（EQ，External Query）： 用户查询系统并获取数据的操作。
  • 为每个功能分配复杂度： 每个功能（数据功能和事务功能）根据其复杂度进行分类。复杂度分类通常为：
  1. 低（Low）： 功能相对简单，处理的元素少。
  2. 中（Average）： 功能中等复杂，处理的元素数量适中。
  3. 高（High）： 功能非常复杂，处理的数据量大，涉及的逻辑复杂。
  • 功能计数项的复杂度等级
  1. 用每个等级的数量*复杂度对应的功能点数，其相加和为UFP
  • TCF：
  • TCF = 0.65 + 0.01(sum(Fi))：Fi：0-5，TCF：0.65-1.35
  • 技术复杂度因子
  • 技术复杂度因子取值范围

6.1.7 NESMA

  • 借鉴IFPUG，并与IFPUG方法兼容
  • 三种类型的功能点估算：
  1. 指示功能点计数：ILF * 35 + EIF * 15
  2. 估算功能点计数：UFP = (7 * ILF + 5 * EIL + 4 * EI + 5 * EO + 4 * EQ)
  3. 详细功能点计数：同IFPUG
  4. 实在不会：凭感觉直接写用例对应的功能点

6.1.8 COSMIC

  • 软件需求理解为由若干个功能过程组成，关注每个功能过程所引起的数据移动，这些数据移动计为功能点。
  • 例如：数据入、数据出、读写数据都算作功能点

6.1.9 GB/T 36964-2018 成本计算模型

  • SDC = (S * PDR) * SWF * RDF * F + DNC
  • 软件研发成本 = （软件规模 * 生产率） * 软件因素调整因子 * 开发因素调整因子 * 人力成本费率 + 直接非人力成本
  • 软件规模估算：功能点估算，例如NESMA
  • 软件规模调整：
  1. S = UFP * GSC * CF
  2. S：调整后的功能点数； UFP：未调整的功能点数； GSC：一般性系统特征调整因子 CF：需求变更调整因子，取值见表参考行业基准数据报告
  • 软件开发工作量估算：
  1. AE = (PDR * S) * SWF * RDF
  2. AE: 调整后的估算工作量，单位为人时； S: 调整后的软件规模，单位为功能点数； PDR: 生产率，单位为人时/功能点，生产率可取软件行业基准数据报告中全行业生产率的50中位值； SWF: 软件因素调整因子，包含业务领域、应用类型及质量特性调整因子，取值详见表； RDF: 开发因素调整因子，包括开发语言、团队经验，取值详见表。
  3. 软件因素调整因子SWF
  1. 开发因素调整因子RDF
  • 软件开发费用估算：
  1. P = \(\frac{AE}{HM2}\) * F2 + DNC
  2. P：软件开发费用，单位为万元。 AE：调整后工作量，单位为人时。 HM2 ：人月折算系数，单位为人时/人月，取值为174(21.75天x8人时/天=174人时)。 F2：人月基准单价，单位为万元/人月，取值见表6-20。 DNC：直接非人力成本，单位为万元；直接非人力成本由委托方根据实际情况进行分析估算。
  • 功能点与代码行转换
  • 实在不会：人力成本150 * 功能点数量

6.1.10 用例点估算法

  • 计算未调整的角色权制UAW
  1. UAW = \(∑_{C=c}aWeight(C) * aCardinality(c)\)
  • 计算未调整的用例权值
  1. UUCW = \(∑_{C=c}uWeight(C) * uCardinality(c)\)
  • 计算未调整的用例点UUCP
  1. UUCP = UAW + UUCW
  • 计算技术和环境因子TEF
  1. 计算技术因子TCF = 0.6 + (0.01 * \(∑^{13}_{i=1}TCF\_Weight_i * Value_i\))
  1. 计算环境因子ECF = 1.4 + (-0.03 * \(∑^8_{i=1}ECF\_Weight_i * Value_i\))
  • 计算调整的用例点UCP
  1. UCP = UUCP * TCP * ECF
  • 计算工作量(man-hours)
  1. Effort = UCP * PF
  2. 一般假设PF = 20工时/用例点

6.1.11 类比(自顶向下)估算法

估算人员根据以往的完成类似项目所消耗的总成本（或工作量），来推算将要开发的软件的总成本（或工作量）

6.1.12 自下而上估算法

利用任务分解图(WBS)，对各个具体工作包进行详细的成本估算，然后将结果累加起来得出项目总成本。

6.1.13 三点估算法

  • 最可能成本（\(C_M\)) :比较现实的估算成本。
  • 最乐观成本（\(C_O\)):最好情况所得到的估算成本。
  • 最悲观成本（\(C_P\)):最差情况所得到的估算成本。
  • 三点估算结果：
  1. 三角分布：\(C_E\) = (\(C_O\) + \(C_M\) + \(C_P\))/3
  2. 贝塔分布：\(C_E\) = (\(C_O\) + 4\(C_M\) + \(C_P\))/6

6.1.14 参数估算法(COCOMO)

  • E = a + b * \(S^c\) * M
  • E：以人月表示的工作量
  • a，b，c：经验导出的系数
  • M：调节因子
  • S：主要的输入参数
  • COCOMO：E = 3.2 *\((KLOC)^{1.05}\) (模型等级：基本、中等、高级)
方式 a b
有机 2.4 1.05
半有机 3.0 1.12
嵌入式 3.6 1.2
  • Walston-Felix(IBM)：E = 5.2 *\((KLOC)^{0.91}\)
  • KLOC：源代码行数
  • a * \(KLOC^b\)

7.1.4 历时估算

  • 估计任务、路径、项目的持续时间
  • 基本方法：
  1. 定额估算法
  2. 经验导出模型
  3. CPM(关键路径法估计)
  4. PERT(工程评估评审技术)
  5. 预留分析

7.1.5 定额估算法

  • T = Q / (R * S)
  • T：活动历时，Q：任务工作量，R：人力数量，S：工作效率

7.1.6 经验导出模型

  • D = a * \(E^b\)
  • D：进度(以月单位)
  • E：工作量(以人月单位)
  • a：2-4之间
  • b：1/3左右(依赖于项目的自然属性)
  1. Walston-Felix模型：D = 2.4 * \(E^{0.35}\)
  2. 基于COCOMO：D = 2.5 * \(E^b\)
方式 b
有机 0.38
半有机 0.35
嵌入式 0.32

7.1.7 CPM(关键路径法)

  • 确定项目网络图
  • 每个任务有单一的历时估算
  • 确定网络图中任务的逻辑关系
  • 关键路径是网络图中最长的路径。
  • 关键路径决定项目完成的最短时间。
  • 关键路径：
  1. 网络图中最长的路径
  2. 关键路径决定项目完成的最短时间。
  3. 关键路径上任何活动延迟，都会导致整个项目完成时间的延迟
  4. 关键路径可能不止一条

7.1.8 PERT(工程评估评审)

  • 选定3个估算值：
  1. 0是最小估算值:乐观(Optimistic)，
  2. P是最大估算值:悲观(Pessimistic)，
  3. M是最大可能估算(Most Likely)。
  • 采用加权平均得到期望值E= O+4m+P)/6
  • 标准差 = (最大估算值-最小估算值)/6

7.1.9 预留分析

  • 应给预留：应急预留是包含在进度基准中的一段储备时间，用来应对已经接受的已识别风险，以应对进度方面的不确定性。
  • 管理预留：管理预留是为管理控制的目的而特别留出的项目预算，用来应对项目范围中不可预见的风险。

7.1.11 网络图

  • PDM(优先图法，节点法)
  1. 构成PDM网络图的基本特点是节点(Box)
  2. 节点(Box)表示活动（任务）
  3. 用箭线表示各活动(任务)之间的逻辑关系
  4. 可以方便的表示活动之间的各种逻辑关系
  • ADM(箭线法)
  1. ADM也称为双代号项目网络图
  2. 在ADM网络图中,箭线表示活动(任务)
  3. 两个代号唯一确定一个任务
  4. 代号表示前一任务的结束,同时也表示后一任务的开始

7.1.12 传统进度编制的基本方法

  • 超前(lead)与滞后(lag)
  • 关键路径法(时间浮动 = 0)
  1. 最早开始时间(Early start)
  2. 最晚开始时间(Late start)
  3. 最早完成时间(Early finish)
  4. 最晚完成时间(Late finish)
  5. 总浮动(Total Float)：在不影响项目最早完成时间的前提下，一个任务可以延迟的时间
  6. 自由浮动(Free Float)：在不影响后置任务最早开始时间的前提下，一个任务可以延迟的时间
  • 正推法：ES + Duration = EF，EF + Lag = ES(s)(选择前置任务中最大的EF加上Lag作为ES)
  • 逆推法：LF - Duration = LS，LS - Lag = LF(p)(选择后置任务中最小的LS减Lag作为LF)
  • 时间压缩法
  1. 在不改变项目范围的前提下缩短项目工期的方法
  2. 应急法：赶工
  3. 平行作业法：快速跟进

7.1.13 资源优化

  • 资源平衡(调整任务时间来协调资源冲突，导致关键路径改变)
  • 资源平滑(不会干扁关键路径)

7.1.14 敏捷计划编排

  • Release planning：发布计划(远期，粗)
  • Iteration planning：迭代计划(近期，细)

11.1 相关定义

11.1.1 风险三要素

  • 风险事件
  • 事件概率
  • 事件影响

11.1.2 风险类型

  • 预测角度：
  1. 已知风险
  2. 可预测风险
  3. 不可预测风险
  • 范围角度

商业风险、管理风险、人员风险、技术风险、开发环境风险、客户风险、过程风险、产品规模风险等。

11.1.3 风险管理的四个过程

  • 风险识别：识别已知和可预测的风险
  • 风险评估：对风险事件发生概率、项目风险影响的评估，给出项目风险排序
  • 风险规划
  • 风险控制：风险控制是在项目执行过程中监控风险，实施风险计划，同时，不断进行风险识别、风险分析、风险规划的过程。

11.1.4 风险评估

  • 风险值：R = F(P，I)
  • 风险发生的概率：P
  • 风险对项目的影响：I

11.1.5 EMV(损益期望值)

EMV = 成功的概率 * 收益

11.1.6 风险规划的主要策略

  • 回避风险：
  1. 回避风险是对可能发生的风险尽可能的规避，采取主动放弃或者拒绝使用导致风险的方案
  2. 例如：放弃采用新技术
  • 转移风险：
  1. 转移风险是为了避免承担风险损失，有意识将损失或与损失有关的财务后果转嫁出去的方法。
  2. 例如：分包，开脱责任合同，保险
  • 损失控制：
  1. 损失预防：项目技术培训，预防技术失败
  2. 损失抑制：项目人员储备，抑制人员流失的损失
  • 自留风险：由项目组织自己承担风险事故所致损失的措施

11.1.7 敏捷项目风险应对方法

  • 损失预防与损失抑制策略：
  1. 跨职能项目团队(识别风险)
  2. 选择迭代内容(选择风险小的)
  3. 频繁评审增量产品
  4. 持续测试可以及早发现问题
  5. 客户参与可以减少需求变更的风险

11.1.8 MVP(最小可行产品)

快速地构建出符合产品预期功能的最小功能集合可以有效降低风险

11.1.9 风险评估方法

  • 模拟
  • 决策树分析
  • 盈亏平衡分析
  • 敏感分析

14.1.3 图解控制法

  • 进度图、成本图、资源图
  • 偏差分析与控制：

  1. 精确记录任务消耗的实际时间

  2. 量化任务的计划偏差：

    持续时间偏差(%) = ((实际持续时间 - 计划持续时间)/ 计划持续时间) * 100

    进度偏差(%) = ((实际结束时间 - 计划结束时间)/ 计划持续时间) * 100

  3. 对计划偏差进行根因分析

14.1.4 挣值分析法

  • 输入：
  1. BAC(预算总值(估算结果))
  2. TAC(预计完成时间)
  3. BCWS(计划工作成本)
  4. ACWP(实际工作成本)
  5. BCWP(已获值(已完成工作的预算成本))
  • 输出：CV，CPI，SV，SPI，EAC，SAC，VAC，VAT，TCPI
  1. 进度差异：SV = BCWP - BCWS(=0：按照计划进度进行，<0：落后于进度，>0：超前于进度)
  2. 成本差异：CV = BCWP - ACWP(=0：按照计划预算进行，<0：比预算差，>0：比预算好)
  3. 进度效能指标：SPI = \(\frac{BCWP}{BCWS}\)(=1：按照计划进度进行，<1：落后于进度，>1：超前于进度)
  4. 成本效能指标：CPI = \(\frac{BCWP}{ACWP}\)(=1：按照计划预算进行，<1：超出预算，>1：低于预算)
  5. 预测项目完成成本：EAC = \(\frac{BAC}{CPI}\)
  6. 预测项目完成时间：SAC = \(\frac{TAC}{SPI}\)
  7. 成本偏差：VAC = BAC - EAC
  8. 时间偏差：VAT = TAC - SAC
  9. 未完工指数：TCPI = \(\frac{剩余工作}{剩余成本} = \frac{(BAC - BCWP)}{(Goal - ACWP)}\)

14.1.5 BCWP的计算

  • 50/50规则：当一项任务开始、没有结束前，获得一半的价值
  • 0/100规则：当一项任务没有完成前，没有价值
  • 经验加权：按照经验百分比计算价值
